Nodes are the building blocks of an agent graph. Each node performs a specific operation and returns a state update that flows into the next node along the graph edges.

All nodes share a common structure in YAML:

```yaml
- id: "unique_node_id"
  type: "node_type"
  parameters:
    # Type-specific parameters
```

<Info>
  The special identifiers `__start__` and `__end__` are reserved for the graph entry and exit points. They are used in edges but never defined as nodes.
</Info>

    ## LLM Node (`node_llm`)

    Calls a large language model with the current conversation history and a system prompt. Optionally binds tools so the LLM can request tool calls.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `model` | string | (required) | Model name or alias (e.g., `gpt-4o`, `fast`, `claude-3-opus`) |
    | `provider` | string | (required) | LLM provider: `litellm`, `openai`, `google`, `anthropic`, `deepseek` |
    | `temperature` | float | `0.0` | Sampling temperature (0.0 = deterministic) |
    | `max_tokens` | integer | `null` | Maximum output tokens |
    | `max_input_tokens` | integer | `8192` | Maximum input context window for history trimming |
    | `prompt_ref` | string | `null` | Key in the `prompts` map to use as the system message |
    | `bind_all_pipelines` | boolean | `false` | Bind every user pipeline as a callable tool (Pipeline-First) |
    | `bind_pipelines` | string[] | `null` | Selectively bind specific pipeline endpoints |
    | `allowed_pipelines` | string[] | `null` | Whitelist of pipeline names this LLM may call |
    | `fallback_models` | string[] | `[]` | Ordered fallback models if the primary fails |
    | `json_mode` | boolean | `false` | Parse the LLM response as JSON |
    | `_internal_only` | boolean | `false` | When `true`, the response is not sent to the user |

    **How it works:**

    1. The message history is sanitized and trimmed to fit within `max_input_tokens`.
    2. The system prompt is rendered from the `prompts` map using Jinja2 templating.
    3. The model is invoked through the configured provider. If the primary model fails, fallbacks are tried in order.
    4. If `bind_all_tools` is enabled, all configured tools are attached to the model.
    5. The model is invoked and the response is added to the message history.
    6. If the response contains tool calls, `has_tool_call` is set to `true` in the context.

    ```yaml
    - id: "main_llm"
      type: "node_llm"
      parameters:
        model: "fast"
        provider: "litellm"
        temperature: 0.2
        max_tokens: 4096
        max_input_tokens: 12000
        prompt_ref: "system_main"
        bind_all_pipelines: true
    ```

    <Tip>
      Use the `fast` model alias for cost-effective operations. The alias resolves to the best available model for your provider configuration. Or use `smart` for top-quality, `cheap` for cost-sensitive, `reasoning` for complex chain-of-thought.
    </Tip>

    <Info>
      **Pipeline-First architecture**: tools are exposed to agents as pipeline endpoints, not as raw integrations. When you set `bind_all_pipelines: true`, every pipeline endpoint defined for the user becomes a callable tool — with its own input/output schema, API key, rate limit, and timeout. Use `bind_pipelines` or `allowed_pipelines` to scope which pipelines a specific LLM call can invoke.
    </Info>

    ## Tool Node (`node_tool`)

    Executes tool calls generated by an LLM node. Routes each call to the appropriate service and pauses the graph until results arrive.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `allowed_tools` | string[] | `[]` | Tool whitelist. Empty list allows all tools. |

    The tool node reads pending tool calls from the state, determines the target for each call (REST API, Python sandbox, data engine, etc.), dispatches the request, and pauses the graph. When the result arrives, the graph resumes and the tool result is added to the message history.

    ```yaml
    - id: "tool_executor"
      type: "node_tool"
      parameters:
        allowed_tools: []
    ```

    See [Tools & Integrations](/agents/tools-and-integrations) for details on tool types and routing.

    ## Knowledge Node (`node_knowledge`)

    Performs a vector similarity search against the knowledge base for retrieval-augmented generation (RAG).

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `collection` | string | `"default"` | Vector collection name |
    | `limit` | integer | `3` | Maximum number of results (1--20) |
    | `threshold` | float | `0.7` | Minimum similarity score (0.0--1.0) |
    | `query_source` | string | `"last_message"` | Source for the query text |
    | `filters` | object | `null` | Additional metadata filters |

    The node performs a semantic search, pauses the graph while results are retrieved, and resumes when they arrive. Search results are stored in `context.rag_results` as formatted text.

    ```yaml
    - id: "search_kb"
      type: "node_knowledge"
      parameters:
        collection: "product_docs"
        limit: 5
        threshold: 0.75
    ```

    ## Approval Node (`node_approval`)

    Human-in-the-loop quality gate with two operating modes.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `mode` | string | `"judge"` | `"judge"` for automatic LLM evaluation, `"always"` for manual human review |
    | `judge_model` | string | `"fast"` | Model used for automatic quality evaluation |
    | `judge_temperature` | float | `0.0` | Judge sampling temperature |
    | `judge_prompt` | string | `null` | Custom evaluation prompt |
    | `approve_threshold` | float | `0.7` | Score at or above this triggers approval |
    | `handoff_threshold` | float | `0.4` | Score below this triggers handoff to a human |
    | `max_revisions` | integer | `2` | Maximum revision cycles before escalation (0--5) |
    | `timeout` | integer | `300` | Approval timeout in seconds (30--3600) |

    **Judge mode:** An LLM evaluates the agent's response and returns a score. Based on the score:
    - Below `handoff_threshold` -- the conversation is handed off to a human agent.
    - Between thresholds -- the response is sent back for revision (up to `max_revisions` times).
    - At or above `approve_threshold` -- the response is approved and sent to the user.

    **Always mode:** The graph pauses and waits for a human staff member to approve, revise, or reject the response.

    ```yaml
    - id: "quality_check"
      type: "node_approval"
      parameters:
        mode: "judge"
        approve_threshold: 0.7
        handoff_threshold: 0.4
        max_revisions: 2
    ```

    ## MCP Server Node (`node_mcp_server`)

    Executes tools from a specific MCP (Model Context Protocol) server.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `server_ref` | string | `""` | MCP server identifier from the integration config |
    | `allowed_tools` | string[] | `[]` | Tool whitelist (empty = all) |
    | `timeout` | integer | `15` | Execution timeout in seconds |

    ```yaml
    - id: "mcp_tools"
      type: "node_mcp_server"
      parameters:
        server_ref: "demo_tools"
        allowed_tools: []
        timeout: 15
    ```

    ## Condition Node (`node_condition`)

    Evaluates an expression and routes the graph to different branches based on the result.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `mode` | string | `"expression"` | Evaluation mode |
    | `expression` | string | (required) | Expression to evaluate |
    | `branches` | object | (required) | Map of result values to target node IDs |

    ```yaml
    - id: "check_report_needed"
      type: "node_condition"
      parameters:
        mode: "expression"
        expression: "state.context.report_format != ''"
        branches:
          "true": "format_report"
          "false": "__end__"
    ```

    ---

    ## Router Node (`node_router`)

    Multi-branch routing based on a state value. Maps specific values to target nodes with a default fallback.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `route_key` | string | (required) | Dot-notation path to the routing value (e.g., `context.detected_intent`) |
    | `routes` | object | (required) | Map of values to target node IDs |
    | `default_route` | string | `"__end__"` | Fallback target if no value matches |

    ```yaml
    - id: "intent_router"
      type: "node_router"
      parameters:
        route_key: "context.detected_intent"
        routes:
          "shipping": "shipping_handler"
          "billing": "billing_handler"
          "technical": "tech_handler"
        default_route: "general_handler"
    ```

    ---

    ## Logic Node (`node_logic`)

    Evaluates a safe expression and stores the result in state. Useful for computed values, counters, and simple transformations.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `expression` | string | `null` | Expression to evaluate (e.g., `retry_count + 1`) |
    | `output_key` | string | `"logic_result"` | State key where the result is stored |

    The expression runs in a sandboxed environment with access to all state variables and context values.

    ```yaml
    - id: "increment_retries"
      type: "node_logic"
      parameters:
        expression: "retry_count + 1"
        output_key: "retry_count"
    ```

    <Note>
      Expressions support basic arithmetic, string operations, comparisons, and structured conditions (EQ, NEQ, GT, LT, CONTAINS, IN_LIST, REGEX, EXISTS). They cannot import modules or call arbitrary functions.
    </Note>

    ## Handover Node (`node_handover`)

    Transfers the conversation from the AI agent to a human staff member. After handover, the AI stops responding until the session is released.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `handoff_message` | string | `"This conversation has been transferred to a representative."` | Message sent to the user |
    | `assign_to_role` | string | `"agent"` | Staff role for assignment |
    | `lock_session` | boolean | `true` | Prevent AI responses after handover |

    ```yaml
    - id: "escalate"
      type: "node_handover"
      parameters:
        handoff_message: "I'm connecting you with a specialist."
        assign_to_role: "senior_agent"
    ```

    ---

    ## Fan-Out Node (`node_fan_out`)

    Publishes messages to multiple targets in a fire-and-forget pattern. Does not pause the graph. Useful for triggering side effects such as archiving data or sending notifications.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `targets` | array | (required) | List of target definitions |
    | `targets[].event_type` | string | (required) | Event type for the published message |
    | `targets[].payload_mapping` | object | `{}` | Map of output keys to state paths |

    ```yaml
    - id: "notify_all"
      type: "node_fan_out"
      parameters:
        targets:
          - event_type: "data.archive"
            payload_mapping:
              entity_name: "literal:customer_interaction"
              data: "context.interaction_summary"
    ```

  <Tab title="Data Nodes">

    ## Set Variable Node (`node_set`)

    Updates a state variable with a static value, a dynamically extracted value, or a computed expression.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `variable` | string | (required) | Target variable name in state |
    | `value` | any | `null` | Static value to assign |
    | `from` | string | `null` | Dot-notation path for dynamic extraction (e.g., `context.messages[-1].content`) |
    | `expression` | string | `null` | Safe expression to evaluate |
    | `operation` | string | `"set"` | Operation: `set`, `increment`, `append` |
    | `validate_schema` | string | `null` | Entity schema name for validation |

    Value resolution priority: `from` > `expression` > `value`.

    ```yaml
    # Static value
    - id: "set_priority"
      type: "node_set"
      parameters:
        variable: "escalation_level"
        value: "high"

    # Dynamic extraction from last LLM response
    - id: "capture_summary"
      type: "node_set"
      parameters:
        variable: "executive_summary"
        from: "context.messages[-1].content"
    ```

    ---

    ## Transform Node (`node_transform`)

    Performs data transformation operations on state values.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `operation` | string | (required) | Operation type (see table below) |

    | Operation | Description | Key Parameters |
    |---|---|---|
    | `extract` | Extract a single field from context or input | `input_key`, `output_key` |
    | `map` | Multi-field mapping with dot-notation references | `mapping: {key: "context.path"}` |
    | `template` | Jinja2 template rendering | `template`, `output_key` |
    | `filter` | Filter a list by condition | `source`, `condition`, `output_key` |
    | `merge` | Merge multiple dicts from state | `sources`, `output_key` |

    **Filter operators:** `==`, `!=`, `in`, `contains`.

    ```yaml
    # Extract a field
    - id: "extract_intent"
      type: "node_transform"
      parameters:
        operation: "extract"
        input_key: "intent"
        output_key: "detected_intent"

    # Map multiple fields
    - id: "build_payload"
      type: "node_transform"
      parameters:
        operation: "map"
        mapping:
          customer_name: "context.user_profile.name"
          order_id: "context.order_details.id"
    ```

    ---

    ## LLM Transform Node (`node_llm_transform`)

    Like `node_transform`, but uses an LLM to perform the transformation. Best for tasks where rules-based mapping is insufficient — fuzzy classification, sentiment, extraction, summarization, language detection.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `model` | string | `"fast"` | Model alias or specific model |
    | `prompt` | string | (required) | Instruction for the LLM |
    | `input_keys` | string[] | (required) | Which state keys to feed into the prompt |
    | `output_key` | string | (required) | Where to store the result |
    | `schema` | object | `null` | JSON schema; if set, output is validated |

    ```yaml
    - id: "classify_intent"
      type: "node_llm_transform"
      parameters:
        model: "fast"
        prompt: "Classify the customer's intent as one of: shipping, billing, technical, sales, other."
        input_keys: ["context.messages[-1].content"]
        output_key: "detected_intent"
        schema:
          type: string
          enum: [shipping, billing, technical, sales, other]
    ```

    ---

    ## PII Scrub Node (`node_pii_scrub`)

    Detects and anonymizes PII (personal information) in state values. Useful for protecting user data before it is logged, archived, or sent to a downstream service.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `input_key` | string | (required) | State path containing the text to scrub |
    | `output_key` | string | (required) | Where to store the scrubbed text |
    | `operation` | string | `"redact"` | One of `redact`, `mask`, `hash`, `replace`, `encrypt` |
    | `entities` | string[] | `auto` | Specific PII types to look for, or `auto` for all |

    See [Privacy & PII](/data/privacy-and-pii) for the full list of detected entities.

    ```yaml
    - id: "scrub_before_archive"
      type: "node_pii_scrub"
      parameters:
        input_key: "context.user_message"
        output_key: "context.user_message_safe"
        operation: "mask"
    ```

    ## Pipeline Node (`node_pipeline`)

    Invokes a pipeline as a tool from within an agent graph. This is the cornerstone of **Pipeline-First architecture** — heavy lifting (data extracts, ML, optimization, scraping) lives in pipelines, while agents call them like tools.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `pipeline_name` | string | (required) | Name of the pipeline endpoint to call |
    | `input_mapping` | object | `{}` | Map state values to pipeline inputs |
    | `output_key` | string | `pipeline_result` | Where to store the result |
    | `sync` | boolean | `true` | Wait for completion; `false` returns immediately with an invocation ID |
    | `timeout` | integer | `120` | Max seconds to wait when `sync: true` |

    ```yaml
    - id: "run_forecast"
      type: "node_pipeline"
      parameters:
        pipeline_name: "weekly_demand_forecast"
        input_mapping:
          horizon_days: "context.requested_horizon"
        output_key: "forecast_result"
        timeout: 60
    ```

    ---

    ## Orchestrator Node (`node_orchestrator`)

    Delegates a sub-task to another agent. The graph pauses while the sub-agent runs and resumes when it returns. Used to build hierarchical agent systems — a generalist routes to specialists who can route to sub-specialists, with no depth limit.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `agent_name` | string | (required) | Target sub-agent |
    | `input_mapping` | object | `{}` | Map state to sub-agent input |
    | `output_key` | string | `sub_result` | Where to store the sub-agent's reply |
    | `preserve_history` | boolean | `false` | Pass full conversation context to the sub-agent |

    See [Multi-Agent Orchestration](/agents/multi-agent-orchestration) for patterns and examples.

    ```yaml
    - id: "billing_specialist"
      type: "node_orchestrator"
      parameters:
        agent_name: "billing_agent"
        input_mapping:
          question: "context.messages[-1].content"
          customer_id: "context.user_profile.id"
        output_key: "billing_reply"
    ```

    ---

    ## DEHA Node (`node_deha`)

    Embed the DEHA meta-agent inside your agent graph. Useful for agents that need to inspect or modify platform configuration on the fly ("create me a new connector," "list my pipelines," "what does this agent cost?").

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `mode` | string | `"chat"` | `chat`, `execute`, `inventory`, or `insights` |
    | `input_mapping` | object | `{}` | Map state to DEHA input |
    | `output_key` | string | `deha_result` | Where to store DEHA's response |
    | `dry_run` | boolean | `true` | For `execute` mode, skip applying changes |

    ```yaml
    - id: "ask_deha"
      type: "node_deha"
      parameters:
        mode: "inventory"
        input_mapping:
          question: "context.messages[-1].content"
        output_key: "platform_summary"
    ```

    ---

    ## Error Recovery Node (`node_error_recovery`)

    Catches unhandled errors from upstream nodes and routes the graph to a recovery flow. Use this to keep agents responsive when tools fail or LLM calls time out.

    **Parameters:**

    | Parameter | Type | Default | Description |
    |---|---|---|---|
    | `recovery_message` | string | `"Sorry, something went wrong. A team member will follow up."` | Fallback reply to the user |
    | `notify` | object | `null` | Optional notification target (Slack channel, email) |
    | `escalate_to_role` | string | `null` | If set, hand off to this human role |

    ```yaml
    - id: "on_error"
      type: "node_error_recovery"
      parameters:
        recovery_message: "I'm having trouble looking that up. A team member will reach out shortly."
        escalate_to_role: "senior_agent"
        notify:
          channel: slack
          target: "#oncall"
    ```

## Node type summary

| Type | YAML value | Category | Description |
|---|---|---|---|
| LLM | `node_llm` | AI | Call a large language model |
| Tool | `node_tool` | AI | Execute LLM-generated tool calls |
| Knowledge | `node_knowledge` | AI | Vector search for RAG |
| Approval | `node_approval` | AI | Human-in-the-loop quality gate |
| MCP Server | `node_mcp_server` | AI | Execute MCP server tools |
| Condition | `node_condition` | Control flow | Branch based on expression |
| Router | `node_router` | Control flow | Multi-branch value-based routing |
| Logic | `node_logic` | Control flow | Evaluate and store expressions |
| Handover | `node_handover` | Control flow | Transfer to human agent |
| Fan-Out | `node_fan_out` | Control flow | Publish to multiple targets |
| Set Variable | `node_set` | Data | Update state variables |
| Transform | `node_transform` | Data | Rule-based data transformation |
| LLM Transform | `node_llm_transform` | Data | LLM-driven transformation (classify, extract, summarize) |
| PII Scrub | `node_pii_scrub` | Data | Detect and anonymize personal information |
| Pipeline | `node_pipeline` | Orchestration | Invoke a pipeline as a tool (Pipeline-First) |
| Orchestrator | `node_orchestrator` | Orchestration | Delegate to a sub-agent |
| DEHA | `node_deha` | Orchestration | Embed DEHA meta-agent in your graph |
| Error Recovery | `node_error_recovery` | Resilience | Catch errors and route to fallback flow |
