DEHA ONE is built with security at its core. Your data is isolated, your access is controlled, and your credentials are encrypted. This page gives you an overview of how DEHA ONE keeps your account safe.

---

## Your data is isolated

Every organization on DEHA ONE operates in its own isolated environment. User isolation is enforced at every layer:

| Layer | Mechanism |
|---|---|
| **Authentication** | JWT carries `user_id`; verified at the Zero-Trust gateway and propagated downstream |
| **Configuration** | Per-user YAML directory; no cross-user config references |
| **Data warehouse** | One DuckDB file per user; no shared connections |
| **MongoDB** | Every query filters on `user_id`; compound indexes enforce it |
| **Vector store** | Per-user Qdrant collections with mandatory `user_id` filter |
| **IoT** | Per-user Redis keys, rate limiters, schemas |
| **Rate limiting** | Per-user counters, per-route, per-entity |
| **Billing** | Independent credit counters, alerts, quotas per user |
| **Vault** | User-scoped key namespace `vault:{user_id}:*` |
| **Streams** | Every event carries `user_id`; routing is user-aware |
| **Lineage** | Marquez namespace enforced as `deha.user.{user_id}`; cross-user emissions dropped |

## How you sign in

DEHA ONE uses **Auth0** for identity and **Single Sign-On (SSO)** for dashboard access.

- Sign in through your company's identity provider (Google, Microsoft, Okta, SAML, OIDC)
- Auth0 JWT (RS256) is verified on every request with a cached JWKS
- Sessions expire automatically after a period of inactivity
- New users are auto-provisioned in your user on first sign-in (configurable per role)

For programmatic access (your apps, CI/CD, integrations), use API keys instead -- see [API Keys](/security/api-keys).

## Team access control

Every team member is assigned a role that determines what they can see and do. DEHA ONE provides five built-in roles, from full administrative access to read-only viewing.

| Role | What they can do |
|------|-----------------|
| **Owner** | Everything, including billing and user management |
| **Admin** | Manage configurations, deploy changes, full operational access |
| **Supervisor** | Edit configurations (cannot deploy to production) |
| **Agent** | Handle conversations and human review tasks |
| **Viewer** | View dashboards and reports (read-only) |

See [Roles and Permissions](/security/roles-and-permissions) for the full breakdown.

## API keys for programmatic access

When you need to connect external systems to DEHA ONE -- such as automation tools, custom applications, or backend services -- you can create API keys from the dashboard. Three scopes are supported:

| Scope | What it can access |
|---|---|
| **User key** | Every agent and pipeline for your user |
| **Per-agent key** | Only one specific agent |
| **Per-pipeline key** | Only one specific pipeline |

Every key:

- Is SHA-256 hashed at rest (the raw key is shown only once on creation)
- Honors per-key sliding-window rate limits
- Supports per-key IP whitelist (user-wide or per-entity)
- Can be rotated or revoked instantly
- Is logged with caller-identity audit on every use

See [API Keys](/security/api-keys) for setup instructions.

---

## IP whitelist

Each user and each agent / pipeline can have an IP whitelist. Whitelisted IPs are stored as exact addresses or CIDR ranges (e.g., `203.0.113.42` or `10.0.0.0/24`). Requests from any other IP are rejected at the gateway.

- User-level whitelist applies to every endpoint for the user
- Per-entity whitelist scopes further (only addresses allowed for that agent / pipeline)
- Max 1000 entries per user (safety cap)
- Can be managed from the dashboard or via API

## How we protect your credentials

All sensitive values -- API tokens, database passwords, OAuth credentials, GCP service-account JSON, certificates -- are stored in an encrypted vault. Your agents and integrations reference these secrets by name, so the actual values never appear in configuration files.

- AES-256-GCM encryption at rest via SOPS + age
- Dual-write to MongoDB for restart resilience
- Vault references resolved at call time, never logged or traced
- Per-user audit log of every vault read

See [BYOK & Vault](/security/byok-and-vault) for the full model, including bringing your own LLM keys.

---

## Encryption

All communication with DEHA ONE is encrypted:

- **In transit**: All traffic uses HTTPS (TLS encryption)
- **At rest**: Sensitive data and secrets are encrypted in storage

---

## Webhook security

If you receive messages from external channels (WhatsApp, Telegram, Email), each channel uses its own signature verification method to ensure messages are authentic and have not been tampered with.
